# AI가 만든 스펙을 어떻게 믿게 만들까 — 구매 결정 서비스 Axis 개발기

> ChatGPT가 쇼핑 리서치까지 해주는 시대에, 굳이 전자제품 구매 비교 서비스를 만들었다.
> "AI 결정"만으로는 챗봇을 못 이긴다는 걸 인정하는 데서 설계가 시작됐다.
> 이 글은 그 해자(moat)를 찾아가는 과정과, 1인 개발의 현실을 정리한 기록이다.

---

## 1. 왜 또 비교 서비스인가

"아이폰 16 vs 갤럭시 S25" 같은 질문은 이미 ChatGPT가 잘 답한다. 9억 명이 쓰는 무료 도구가 하는 일을 똑같이 만드는 건 의미가 없다. 그래서 첫 질문을 뒤집었다.

**"챗봇이 구조적으로 못 하는 게 뭔가?"**

세 가지로 좁혀졌다.

1. **실시간 가격** — LLM은 어제의 가격도 모른다.
2. **검증된 스펙** — LLM은 그럴듯한 숫자를 지어낸다(환각).
3. **가격 알림** — 챗봇은 내가 산 뒤 가격이 떨어졌는지 알려주지 않는다.

이 셋을 묶어 제품 한 줄 정의를 잡았다.

> **결정(Decide) → 추적(Track) → 알림(Alert)**
> 사기 전에 결정해주고, 산 뒤 최적 타이밍까지 알려주는 도구.

기술 스택은 Next.js 16(App Router) / React 19 / Supabase / Vercel. AI는 Groq(Llama 3.1), 가격은 네이버 쇼핑 API. 핵심은 스택이 아니라 **AI의 출력을 어떻게 신뢰 가능한 정보로 만드느냐**였다.

---

## 2. 핵심 설계 — 검증 게이트(Verification Gate)

AI 비교 서비스의 진짜 적은 경쟁사가 아니라 **환각**이다. "iPhone 16 Pro의 주사율은 60Hz"처럼 자신만만하게 틀린 답을 내면, 서비스 신뢰도는 한 번에 무너진다.

그래서 모든 비교 결과에 **출처 등급**을 매기고, 등급에 따라 노출·색인 정책을 다르게 가져갔다.

| 등급 | 조건 | SEO |
|------|------|-----|
| `verified` | 모든 primary 스펙에 공식 소스 URL이 붙음 | ✅ 인덱싱 |
| `partial` | 일부 primary 스펙만 출처 보유 | 조건부 |
| `unverified` | AI 단독 생성 (출처 없음) | ❌ noindex |

출처 티어는 **1 제조사 공식 / 2 검증 자료 / 3 AI 추정**으로 나누고, **2티어 이상일 때만 `verified` 배지**를 준다. AI 추정값은 검색엔진에 색인하지 않는다. "없는 제품은 그럴듯하게 지어내지 말고 '찾을 수 없음'으로 떨어뜨린다"를 절대 규율로 삼았다.

이건 제품 철학이자 코드 규율이다. 데이터셋은 손으로 검증한 122개 제품(스마트폰·노트북·이어폰·태블릿)을 하드코딩으로 들고 있고, 스펙은 DB가 아니라 이 검증 데이터셋 또는 공식 페이지 파싱에서만 가져온다.

## 3. 아키텍처 — 두 번의 추상화

비교 요청 흐름(`lib/decision-engine.ts`)은 다음과 같다.

```
사용자 쿼리 ("에어팟 프로 vs 버즈")
  ↓
[1] 캐시 확인 (Supabase, key = v{버전}|query|locale|country)
[2] 브랜드명 → 최신 모델 확장 / 카테고리 분류
[3] 공식 소스 URL 확인 (registry → alias → URL)
[4] 공식 페이지 스크래핑 (병렬) + 검증 데이터셋 fallback
[5] AI 결정 (Groq) — 스키마 필드 전체 채우기
[6] 스크래핑 값 오버레이 + source URL 부착(검증 배지)
[7] 등급 판정 → 캐시 저장
```

설계에서 가장 잘한 결정은 **두 개의 프로바이더 추상화**다.

- **AI 프로바이더** — Groq / OpenAI / Gemini / Anthropic을 한 인터페이스 뒤에 두고 환경변수로 교체. 무료 티어가 막히면 다른 곳으로 폴백할 길을 열어뒀다.
- **가격 프로바이더** — `AXIS_PRICE_SOURCE` 한 줄로 `naver` / `coupang` / `seed`를 전환. 쿠팡 파트너스 정식 승인을 기다리는 동안 네이버로 운영하다가, 승인되면 **환경변수만 바꿔** 전환할 수 있다.

추상화는 "혹시 몰라서"가 아니라, **1인 개발에서 외부 의존성의 불확실성(승인 대기, rate limit, 가격 정책)을 코드 변경 없이 흡수하기 위한** 실용적 장치였다.

## 4. 가장 오래 잡았던 버그 — 스크래핑이 검증 데이터를 덮어쓰다

증상은 단순했다. **"아이폰 16 프로"의 주사율이 60Hz로 표시**됐다(실제 120Hz). 더 심한 케이스로는 아이폰 12 Pro가 비교표에서 "Apple Pencil(2세대)"로 둔갑하기도 했다.

근본 원인은 merge 순서였다.

```ts
// 이전 — 스크래핑이 성공하면 검증 데이터셋을 통째로 무시
if (hasDataset && !scraped) return datasetEntry;
return { ...datasetSpecs, ...spec.specs }; // scraped가 dataset을 덮어씀
```

Apple 공식 페이지는 주사율을 "ProMotion"이라는 산문으로 표현한다. 정적 스크래퍼가 `refresh_hz` 필드를 못 뽑고, 빈 값을 AI가 채우는 과정에서 다른 모델 값과 뒤섞였다. **라이브 스크래핑이 손으로 검증한 데이터보다 우선**이라는 가정이 틀렸던 것이다.

수정은 우선순위를 뒤집는 것이었다.

```ts
// 이후 — 검증 데이터셋이 항상 이긴다. 스크래핑은 빈 필드만 보완
const mergedSpecs = { ...spec.specs, ...datasetSpecs };
return { ...spec, source: entry.source, specs: mergedSpecs };
```

교훈은 명확했다. **"최신 데이터가 가장 정확하다"는 직관은, 소스의 신뢰도가 보장될 때만 옳다.** 자동 수집은 조용히 틀리고, 손으로 검증한 데이터는 틀려도 추적 가능하다. 신뢰의 우선순위를 코드에 박아 넣는 것이 검증 게이트의 실체였다.

(스키마·결과 포맷이 바뀔 때마다 `CACHE_VERSION`을 올려 구버전 캐시를 무효화하는 규율도 이때 자리 잡았다. 현재 v9.)

## 5. 맞춤 재분석 — 같은 제품, 다른 결론

스펙 우위만으로 고른 "객관적 정답"은 종종 사용자에게 정답이 아니다. 게임이 중요한 사람과 휴대성이 중요한 사람의 답은 다르다. 그래서 **사용자 상황을 결론에 주입하는 맞춤 재분석**을 붙였다.

용도·예산·자유 메모를 받아 `userContext`로 만들고, 이를 AI 프롬프트 최상단에 우선순위로 올린다.

```ts
const userContextBlock = input.userContext?.trim()
  ? `User's situation (THIS IS THE PRIORITY — tailor the verdict to it):
"${input.userContext.trim()}"
- 원시 스펙에서 다른 선택지가 이겨도, 이 사용자에게 맞는 쪽으로 결론을 기울인다.
- reasons/analyses에서 그 선택을 사용자의 니즈와 명시적으로 연결한다.`
  : "";
```

핵심 디테일은 **캐시 우회**였다. 일반 비교는 24시간 캐시로 토큰을 아끼지만, 맞춤 결과는 사람마다 달라야 하므로 캐시를 읽지도 쓰지도 않는다.

```ts
const tailored = Boolean(userContext?.trim());
if (!tailored) {
  const cached = await getCachedComparison(query, locale, country);
  if (cached) return cached;
}
// ... 결과 저장도 동일하게 가드
if (!tailored) void setCachedComparison(query, locale, country, finalResult);
```

실제로 "맥북 에어 vs 프로"에 "게임 / 100~200만원"을 넣으면 결론이 **"게임을 즐기기에는 OLED·120Hz의 맥북 프로가 가장 적합"**으로 바뀐다. 같은 입력, 다른 맥락, 다른 답.

추천 문구도 함께 손봤다. 스펙 숫자를 나열하지 말고 **실생활 의미로 번역**하도록 프롬프트를 강화했다. "배터리 5000mAh"가 아니라 "헤비하게 써도 하루를 거뜬히 넘긴다"로.

## 6. 현실의 벽 — 무료 티어로 운영하기

검증 단계에서 직접 만난 제약은 **Groq 무료 티어의 분당 토큰 한도(6,000 TPM)**였다. 프롬프트가 호출당 약 2,600토큰이라, 캐시를 우회하는 맞춤 재분석을 몇 번 연속 호출하면 곧바로 HTTP 429가 떨어진다.

다행히 구조가 이를 흡수한다.

- **24시간 캐시**가 인기 비교(수십 가지로 수렴)를 대부분 흡수한다. 토큰을 쓰는 건 트래픽 총량이 아니라 **고유 쿼리 수**다.
- 429가 나면 조용히 **결정론적 비교(deterministic fallback)**로 떨어져, 서비스가 죽지 않는다.
- 트래픽이 커지면 AI 프로바이더 폴백 체인(무료 티어 합산) → Groq 유료(호출 1,000건당 $0.2 미만) 순으로 올라가면 된다.

결론적으로 **AI 비용은 끝까지 병목이 아니다.** 진짜 제약은 돈이 아니라 무료 티어의 처리량이고, 그 시점이 오면 이미 제휴 수익이 비용을 덮는다. "지금 트래픽이 없을 때는 무료로 충분하다"는 판단을 데이터로 확인한 셈이다.

## 7. 냉정한 회고 — 34점짜리 사업성

개발기는 보통 성공으로 끝나지만, 이 프로젝트는 스스로 **투자 심사 관점의 사업성 검증**을 거쳤고 결과는 **종합 34/100, 조건부 GO(Conditional GO)**였다.

냉정하게 드러난 사실들:

- 방어력이 걸린 "라이브 가격+알림"의 경제성이 아직 미검증이다. 외부 가격이력 API는 전부 고비용이거나 닫혀 있다.
- 완성된 "AI 결정"은 ChatGPT의 무료 쇼핑 리서치가 잠식한다.
- 유입 통로인 SEO는 Google AI Overview가 클릭률을 반토막낸다.
- 전자제품 제휴 수수료는 최저(1~3%)이고 구매 빈도도 낮다.

그래서 방향을 좁혔다. **글로벌 + 4개 카테고리 + 푸시·구독**이라는 초기 욕심을 버리고, **한국 · 노트북 · 제휴**라는 단일 슬라이스로 축소해 핵심 가정(데이터 확보 → 유입 → 제휴 전환)부터 현금 0원으로 검증하기로 했다. 통과하지 못하면 결정 전용으로 피벗하거나 멈춘다.

1인 개발에서 가장 비싼 자원은 시간이다. **"무엇을 만들까"보다 "무엇을 만들지 않을까"를 정하는 것**이 이 단계의 핵심 의사결정이었다.

---

## 마치며

Axis는 화려한 기술 스택의 프로젝트가 아니다. 오히려 **AI의 출력을 어디까지 믿고, 어디서 끊을 것인가**라는 신뢰의 경계를 코드로 그어가는 작업이었다. 검증 게이트, merge 우선순위, 캐시 우회, noindex 정책 — 전부 같은 질문의 다른 답이다.

그리고 그 위에서, 만든 것을 냉정하게 평가하고 범위를 줄이는 일까지가 개발이라는 걸 배웠다. 코드를 늘리는 것보다 어려운 건, 만들지 않기로 결정하는 일이었다.

> 코드: 검증 게이트 · 프로바이더 추상화 · 맞춤 재분석은 `lib/decision-engine.ts`와 `lib/ai/`에 있다.
> 운영 데모: https://axis-app-beta.vercel.app
